Diagramme de séquence

Le diagramme de séquence s’attache à modéliser l’aspect dynamique du système. Il peut être comparé à un storyboard mettant en évidence les interactions existantes entre des objets du système. L’accent est mis sur l’ordre chronologique des messages émis, l’axe du temps étant l’axe vertical dans ce diagramme et les premiers messages émis étant situés graphiquement le plus haut.

Pour chaque action de l’utilisateur définie dans le diagramme de cas d’utilisation, il convient d’établir un diagramme de séquence. La fonction de lecture de tous les clients est utilisée ici comme exemple.

 Dans la vue Model Explorer, sélectionnez le modèle racine, faites un clic droit sur l’élément et choisissez l’option New Diagram - Create a new UML Sequence Diagram.

 Choisissez un nom pour ce diagramme, comme par exemple SequenceLectureClientsGlobal.

Le diagramme est généré avec une autre palette.

 Faites un clic droit sur l’élément model de la vue Model Explorer et ajoutez un package avec l’option New Child - Package. Appelez cet élément dialogue. 

 Répétez l’opération pour créer un package contrôle.

 Faites un clic droit sur ce package dans la même vue et ajoutez une classe avec l’option New Child - Class. Appelez cet élément VueClients.

Répétez l’opération pour les classes que vous voulez introduire dans le modèle. 

 Utilisez les éléments présents dans la vue Model Explorer pour bâtir le diagramme en y faisant glisser des éléments interagissant entre eux.

 Utilisez la section Edges de la palette pour créer des flèches représentant les interactions entre les éléments.

Les flèches ne peuvent se relier aux composants du système que sur un rectangle présent dans leur ligne de vie.

 Pour ajouter ce rectangle, sélectionnez l’élément Action Execution Specification dans la palette à la section Nodes, et faites-le glisser sur l’élément voulu.

Le premier diagramme présenté ci-après représente une première itération possible.

Une conception du système en langage-objet comme Java nécessite pour la clarté et la maintenance de l’application de ne confier idéalement qu’une seule responsabilité par classe. Le contrôle client a ici trop de responsabilités à gérer, certaines de ces responsabilités sont communes avec le futur contrôle article, il est donc avantageux de le découper en plus petites classes.

Une deuxième itération dans la construction du diagramme permet de mieux cerner les différents éléments nécessaires aux interactions entre le contrôle client et la base de données.
