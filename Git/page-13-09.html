Mise en ligne du dépôt sur GitHub

Pour que les lecteurs de ce livre puissent télécharger facilement un dépôt exploitable et suivre l’évolution de ce projet, il est préférable d’héberger également ce dépôt sur GitHub. Dans un scénario plus réaliste, cette partie pourrait expliquer comment une entreprise peut libérer le code source d’un de ses projets. Cette partie va expliquer les étapes pour mettre en ligne un projet sur GitHub en tenant compte des bonnes pratiques.

1. Création d’un compte GitHub

Pour publier un dépôt sur GitHub, il faut un compte GitHub. Pour cela, il faut se rendre sur le site http://github.com et cliquer sur le lien Sign up en haut de la page. Un formulaire s’affiche alors qui demande un nom d’utilisateur Username, une adresse e-mail Email Address et un mot de passe Password.

Après avoir rempli ces informations et cliqué sur le bouton Create an account, il faut choisir le type de compte à enregistrer (le plan gratuit ne permettant de mettre en ligne que des dépôts publics) puis cliquer sur Finish sign up.

Une fois connecté sur le compte GitHub, il est possible d’accéder à son profil en cliquant en haut à droite sur son avatar puis en cliquant sur Your Profile.

2. Création d’un dépôt

À partir de la page de profil, il est possible d’afficher la liste des dépôts de l’utilisateur (vide au début) en cliquant sur l’onglet Repositories. Cette nouvelle vue permet de créer un nouveau dépôt en cliquant sur le bouton vert New.

Un formulaire s’affiche et demande plusieurs informations. L’information Owner est utile pour les utilisateurs qui possèdent un compte avec plusieurs profils. Dans Repository name il faut définir le nom du dépôt, ici Improllow-up.

Il est ensuite possible de définir une description du dépôt en quelques mots, ici Logiciel de suivi de projet et dépôt d’exemple du chapitre 11 du livre Git - Maîtriser la gestion de vos versions.

Il faut ensuite définir la visibilité du projet, ici Public.

Les champs restants permettent de créer automatiquement un fichier README pour le projet, un fichier .gitignore ou un fichier LICENSE. Cette fonctionnalité permet ensuite de cloner le dépôt créé car ces fichiers seront stockés dans un premier commit. Dans cet exemple, aucun fichier ne sera ajouté, car le but est d’importer un dépôt existant.

Le dépôt est ensuite créé après avoir cliqué sur le bouton Create Repository. Pour le moment, le dépôt est vide.

3. Ajout du remote au dépôt local

Le dépôt créé sur le compte GitHub est vide. Il va falloir lui envoyer les données du dépôt local. Pour cela, un développeur, Raphaël par exemple, doit se placer sur un dépôt local à jour. Il récupère l’URL de son dépôt GitHub, à savoir https://github.com/SamuelDauzon/Improllow-up.git.

Cette URL va lui permettre d’ajouter un remote à son dépôt avec la commande suivante : 

git remote add github https://github.com/SamuelDauzon/Improllow-up.git

4. Envoi des branches

Une fois le remote ajouté, Raphaël peut l’utiliser pour envoyer les données de ses branches master et develop avec les commandes suivantes : 

git push -u github master    git push -u github develop    git push github --tags

5. Le fichier LICENSE

Lorsqu’un projet est mis en ligne sur GitHub, il convient de choisir une licence libre qui permettra de limiter ou de libérer les droits des utilisateurs à utiliser, modifier ou vendre le logiciel. Les licences sont caractérisées par leur permissivité ou encore par la clause de copyleft. Par exemple, la licence GPL contient la clause copyleft, ce qui signifie qu’un logiciel incluant du code GPL devient forcément lui aussi GPL. Au contraire, du code sous licence MIT (non copyleft) peut être utilisé dans du code propriétaire.

Raphaël ajoute donc le fichier LICENSE au dépôt en utilisant la licence MIT.

6. Le fichier README

Lorsqu’un projet est hébergé sur GitHub, il est consultable par beaucoup d’internautes. Certains ne sont pas forcément familiers avec la technologie utilisée (ici Django), mais veulent malgré tout tester le logiciel. D’autres voudront participer au projet, mais ne sauront pas comment faire. La rédaction du fichier README permettra de répondre à ces différentes questions.

Le début d’un fichier README doit présenter le projet. Lorsqu’une personne consultera la page GitHub du projet, le premier contenu qu’il lira sera le contenu du fichier README. Il faut donc y expliquer le projet et son utilité.

Le début du fichier README sera donc le suivant :

# Improllow-up    ## Présentation   Ce logiciel permet de **suivre les tâches des membres d’une   équipe**. Ces tâches sont groupées par projets, eux-mêmes groupés   par client.   Ce projet permet de **savoir facilement les projets sur lesquels   a travaillé un collaborateur** sur une période donnée. Un export   CSV des tâches est également présent.   ## Technologie   Ce projet utilise le framework Django (Python) pour la partie   serveur. Par défaut, les données sont stockées dans une base de   données SQLite. Il est possible d’utiliser un autre système de   bases de données en personnalisant la configuration du projet   Django.

Dans la suite du fichier, il s’agit d’expliquer comment utiliser l’application.

Pour ne pas dupliquer de contenu, les actions permettant d’utiliser Improllow-up ne seront pas réécrites, car elles sont présentes dans ce chapitre à la section Aperçu du projet.

La dernière partie figurant dans un fichier README permet de définir des règles simples de participation au projet. Certains projets comme Bower adoptent des règles particulièrement strictes qui permettent de mieux gérer les demandes de contributions : https://github.com/bower/bower/blob/master/CONTRIBUTING.md.

Il est possible de s’inspirer des règles de contributions des dépôts populaires pour mettre en place ses propres normes. Pour les projets très simples et peu populaires, il n’est pas forcément nécessaire d’ériger une liste de règles strictes. Parfois, toutes les contributions sont bonnes à prendre.
